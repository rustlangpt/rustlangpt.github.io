Cargo: O Gestor de Pacotes do Rust

Um guia completo sobre o Cargo: instalação, o ficheiro Cargo.toml, criação de pacotes, gestor de pacotes, gestão de dependencias, Testes e CI/CD.

RustLangPT publicado em 12 September, 2025
5 min, 941 words

Tags: cargo setup Iniciantes

Cargo é o gestor de pacotes oficial da linguagem de programação Rust, responsável por tarefas como descarregar dependências, compilar pacotes, criar pacotes distribuíveis e carregá-los para o crates.io, o registo de pacotes da comunidade Rust. Desenvolvido para simplificar o desenvolvimento em Rust, Cargo integra-se perfeitamente com o ecossistema da linguagem, permitindo uma gestão eficiente de projetos.

Neste artigo, vamos desvendar os segredos do Cargo, desde a sua instalação até às suas funcionalidades mais avançadas, sempre com a documentação oficial do Cargo como nosso guia.

Processo de Instalação

Instalação Automática com Rustup

O Cargo é instalado automaticamente quando instala o Rust através do rustup. Esta é a forma recomendada e mais simples:

# Instalar Rust e Cargo via rustup
$ curl --proto '=https' --tlsv1.2 https://sh.rustup.rs -sSf | sh

# Verificar instalação
$ cargo --version
cargo 1.75.0 (1d8b05cdd 2023-11-20)

Verificação da Instalação

Para confirmar que o Cargo está correctamente instalado e configurado:

# Verificar versão do Cargo
$ cargo --version

# Ver informação detalhada sobre a instalação
$ cargo version --verbose
cargo 1.75.0 (1d8b05cdd 2023-11-20)
release: 1.75.0
commit-hash: 1d8b05cdd2353964c50fe0b8f07a16e4c8caaf15
commit-date: 2023-11-20
host: x86_64-unknown-linux-gnu

# Testar funcionalidades básicas
$ cargo help

Atualização do Cargo

O Cargo é atualizado automaticamente quando atualiza o Rust:

# Atualizar Rust e Cargo
$ rustup update

Criação de Pacotes

Criação de um Novo Projeto

O comando cargo new é a forma padrão de criar novos projetos Rust:

# Criar aplicação executável (binary)
$ cargo new minha-app
Created binary (application) `minha-app` package

# Criar biblioteca (library)
$ cargo new minha-lib --lib
Created library `minha-lib` package

# Criar projeto no diretório atual
$ cargo init

# Criar com VCS específico, por exemplo o git
$ cargo new meu-projeto --vcs git

Isto cria uma pasta minha-app com a seguinte estrutura:

minha-app/
├── src/
│   └── main.rs  (Onde o teu código vai estar)
└── Cargo.toml   (O ficheiro de configuração do teu pacote)

No caso da biblioteca, a estrutura é semelhante, mas o ficheiro principal é src/lib.rs.

Estrutura de um Projeto Cargo

Um projeto típico criado pelo Cargo tem esta estrutura:

minha-app/
├── Cargo.toml        # Manifest do projeto
├── Cargo.lock        # Lock file das dependências
├── src/              # Código fonte
│   ├── main.rs       # Ponto de entrada (para binários)
│   └── lib.rs        # Módulo raiz (para bibliotecas)
├── tests/            # Testes de integração
├── examples/         # Exemplos de uso
├── benches/          # Benchmarks
└── target/           # Diretório de build (gerado automaticamente)

O Ficheiro Cargo.toml

O ficheiro Cargo.toml é a alma do teu projeto Rust. É nele que defines os metadados do teu pacote e, mais importante, as tuas dependências.

O ficheiro Cargo.toml é o manifesto do projeto, seguindo o formato TOML (exemplo):

[package]
name = "meu_projeto"
version = "0.1.0"
edition = "2021"
authors = ["Seu Nome <email@example.com>"]
description = "Uma breve descrição do projeto"
license = "MIT OR Apache-2.0"
repository = "https://github.com/user/projeto"
homepage = "https://projeto.example.com"
documentation = "https://docs.rs/meu_projeto"
readme = "README.md"
keywords = ["web", "api", "rust"]
categories = ["web-programming"]

[dependencies]
serde = { version = "1.0", features = ["derive"] }
tokio = { version = "1.0", features = ["full"] }

[dev-dependencies]
criterion = "0.4"

[build-dependencies]
cc = "1.0"

Para aprenderes mais sobre o formato e as opções do Cargo.toml, a documentação oficial é o teu melhor recurso.

Gestão de Pacotes

A gestão de pacotes no Cargo envolve comandos para construir, executar, atualizar e instalar pacotes. Embora a documentação de comandos gerais esteja indexada, focamo-nos em comandos chave para gestão.

Comandos Principais:

  • Construir: cargo build – Compila o pacote atual.
  • Executar: cargo run – Compila e executa o binário principal.
  • Atualizar Dependências: cargo update – Atualiza as dependências no Cargo.lock.
  • Instalar Pacotes Globais: cargo install nome_do_crate – Instala um crate como ferramenta global.
  • Procurar Pacotes: cargo search nome – Procura crates no crates.io.

Comandos Fundamentais de Build

# Compilar o projeto
$ cargo build
   Compiling minha-app v0.1.0 (/path/to/minha-app)
    Finished dev [unoptimized + debuginfo] target(s) in 2.41s

# Compilar em modo release (otimizado)
$ cargo build --release

# Compilar e executar
$ cargo run
    Finished dev [unoptimized + debuginfo] target(s) in 0.0s
     Running `target/debug/minha-app`
Hello, world!

# Verificar compilação sem gerar executável
$ cargo check

Gestão de Builds e Cache

Em comandos subsequentes $ cargo build, o Cargo não recompila a crate porque não há mudanças no código fonte, dependências ou configurações de build. O Cargo também compila e faz cache das dependências dentro do diretório target/debug/deps.

# Limpar artefactos de build
$ cargo clean

# Build com informação detalhada
$ cargo build --verbose

# Build para targets específicos
$ cargo build --target x86_64-pc-windows-gnu

Gestão de Dependências

Adicionar Dependências

crates.io é o registo central de pacotes da comunidade Rust que serve como local para pesquisar e descarregar pacotes. O cargo está configurado para usá-lo por defeito para encontrar pacotes solicitados. Para depender de uma biblioteca alojada em crates.io, adicione-a ao seu Cargo.toml.

Dependências Básicas

[dependencies]
serde = "1.0"
tokio = { version = "1.0", features = ["full"] }
reqwest = { version = "0.11", features = ["json"] }

# Dependências de desenvolvimento (apenas para testes)
[dev-dependencies]
tokio-test = "0.4"

# Dependências de build (para build scripts)
[build-dependencies]
cc = "1.0"

Especificação de Versões

A string de versão "1.0.136" é chamada requirement de versão.

[dependencies]
# Versão exata
serde = "=1.0.136"

# Compatibilidade com patch
serde = "1.0.136"  # >=1.0.136, <1.1.0

# Compatibilidade com minor
serde = "1.*"      # >=1.0.0, <2.0.0

# Range de versões
serde = ">=1.0.100, <1.2.0"

# Versão mais recente
serde = "*"

# Pre-release
serde = "1.0.0-alpha"

Comando cargo add

Adicionando via cmds

# Adicionar dependência
$ cargo add serde
    Updating crates.io index
      Adding serde v1.0.193 to dependencies.

# Adicionar com features
$ cargo add tokio --features full

# Adicionar dependência de desenvolvimento
$ cargo add --dev tokio-test

# Adicionar dependência opcional
$ cargo add --optional serde

Atualização de Dependências

# Atualizar todas as dependências
$ cargo update

# Atualizar dependência específica
$ cargo update serde

# Atualizar para versão específica
$ cargo update serde --precise 1.0.150

Publicação de Pacotes

Quando a tua biblioteca estiver pronta para ser partilhada com o mundo, o Cargo torna o processo de publicação no crates.io incrivelmente simples.

Passos que deves seguir:

  • Criar uma conta em crates.io.
  • Obter token de acesso no crates.io
  • Fazer login no terminal.
  • Publicar.

Preparação para Publicação

Antes de publicar no crates.io, certifique-se que tem os metadados necessários no Cargo.toml:

[package]
name = "minha-biblioteca"
version = "0.1.0"
edition = "2021"
authors = ["Seu Nome <email@example.com>"]
license = "MIT"
description = "Uma biblioteca útil para demonstração"
homepage = "https://github.com/usuario/minha-biblioteca"
repository = "https://github.com/usuario/minha-biblioteca"
documentation = "https://docs.rs/minha-biblioteca"
readme = "README.md"
keywords = ["rust", "biblioteca", "exemplo"]
categories = ["development-tools"]

[dependencies]
# suas dependências aqui

Criar Conta no Crates.io

  1. Visite crates.io e crie conta via GitHub
  2. Obtenha o seu API token no Account Settings
  3. Configure o token localmente no terminal:
$ cargo login
# Cole o token quando solicitado

Processo de Publicação

Para isso usamos o subcomando 'cargo publish'. Este comando executa os seguintes passos: executa algumas verificações de validação no seu pacote, comprime o seu código fonte num ficheiro '.crate', extrai o ficheiro '.crate' para um diretório temporário e verifica que compila.

Empacotamento e Verificação

# Criar pacote local para verificação
$ cargo package
   Packaging minha-biblioteca v0.1.0 (/path/to/projeto)
   Verifying minha-biblioteca v0.1.0 (/path/to/projeto)
   Compiling minha-biblioteca v0.1.0 (/path/to/projeto/target/package/minha-biblioteca-0.1.0)
    Finished dev [unoptimized + debuginfo] target(s) in 0.61s

# Verificar conteúdo do pacote
$ cargo package --list

Como bónus adicional, o *.crate será verificado independentemente da árvore fonte atual. Após a criação do *.crate, é desempacotado em target/package e depois construído do zero para garantir que todos os ficheiros necessários estão presentes para que o build tenha sucesso.

Publicação

Para a publicação basta fazer o comando cargo publish.

# Publicar no crates.io
$ cargo publish
    Updating crates.io index
   Packaging minha-biblioteca v0.1.0 (/path/to/projeto)
   Verifying minha-biblioteca v0.1.0 (/path/to/projeto)
   Compiling minha-biblioteca v0.1.0 (/path/to/projeto/target/package/minha-biblioteca-0.1.0)
    Finished dev [unoptimized + debuginfo] target(s) in 0.61s
   Uploading minha-biblioteca v0.1.0 (/path/to/projeto)

# Dry-run para testar sem publicar
$ cargo publish --dry-run

Versionamento Semântico

O Cargo segue o Semantic Versioning:

  • MAJOR.MINOR.PATCH (e.g., 1.2.3)
  • MAJOR: Breaking changes
  • MINOR: Nova funcionalidade (backward compatible)
  • PATCH: Bug fixes

Se publicar uma versão com problemas:

# Remover versão problemática (impede novos usos)
$ cargo yank --version 0.1.1

# Reverter yank
$ cargo yank --version 0.1.1 --undo

Testes

O Cargo tem suporte para testes, não precisando de frameworks de terceiros para começar a testar o teu código.

Tipos de Testes:

  • Unitários: Em ficheiros src/ com #[test].
  • Integração: Em tests/nome_teste.rs, importando o crate.
  • Documentação: Em comentários com /// # Examples.

Testes Unitários

// src/lib.rs
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_add() {
        assert_eq!(add(2, 2), 4);
    }

}

Testes de Integração

// tests/integration_test.rs
use meu_projeto::add;

#[test]
fn test_integration_add() {
    assert_eq!(add(5, 5), 10);
}

Doc Tests

/// Adiciona dois números.
///
/// # Examples
///
/// ```
/// use meu_projeto::add;
/// 
/// let result = add(2, 3);
/// assert_eq!(result, 5);
/// ```
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

Comandos de Teste

# Executar todos os testes
$ cargo test
   Compiling meu_projeto v0.1.0 (/path/to/meu_projeto)
    Finished test [unoptimized + debuginfo] target(s) in 0.61s
     Running unittests src/lib.rs (target/debug/deps/meu_projeto-...)
     Running tests/integration_test.rs (target/debug/deps/integration_test-...)

# Executar testes específicos
$ cargo test test_add

# Executar testes ignorados
$ cargo test -- --ignored

# Executar testes com output detalhado
$ cargo test -- --nocapture

# Executar testes em série (não paralelo)
$ cargo test -- --test-threads=1

# Executar apenas unit tests
$ cargo test --lib

# Executar apenas integration tests
$ cargo test --test integration_test

# Executar apenas doc tests
$ cargo test --doc

Configuração de Testes

# Cargo.toml
[dependencies]
serde = { version = "1.0", features = ["derive"] }

[dev-dependencies]
tokio-test = "0.4"
criterion = "0.5"
proptest = "1.0"
mockall = "0.11"
serial_test = "2.0"

# Configuração específica para testes
[[test]]
name = "integration"
path = "tests/integration/main.rs"
required-features = ["integration"]

Benchmarks

Para benchmarks, adicione ao Cargo.toml:

[dev-dependencies]
...

[[bench]]
name = "meus_benchmarks"
harness = false

// benches/benchmark.rs
use criterion::{black_box, criterion_group, criterion_main, Criterion};
use meu_projeto::add;

fn criterion_benchmark(c: &mut Criterion) {
    c.bench_function("add 2 + 2", |b| b.iter(|| add(black_box(2), black_box(2))));
}

criterion_group!(benches, criterion_benchmark);
criterion_main!(benches);

# Executar benchmarks
$ cargo bench
   Compiling meu_projeto v0.1.0 (/path/to/meu_projeto)
    Finished bench [optimized] target(s) in 0.88s
     Running benches/benchmark.rs (target/release/deps/benchmark-...)
add 2 + 2               time:   [0.0000 ns 0.0001 ns 0.0002 ns]

Continuous Integration

GitHub Actions

Verificar as versões mais recentes ajudaria... cargo add ou cargo install. Ao testar as versões mais recentes algumas considerações são: minimizar fatores externos que afetam o desenvolvimento local ou CI, custos de CI, incluindo custos indiretos como se um serviço de CI tem um máximo de runners paralelos, fazendo com que novos jobs sejam serializados quando no máximo.

Configuração Básica

# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

env:
  CARGO_TERM_COLOR: always

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        rust-version: [stable, beta, nightly]
        
    steps:
    - uses: actions/checkout@v4
    
    - name: Install Rust
      uses: dtolnay/rust-toolchain@stable
      with:
        toolchain: ${{ matrix.rust-version }}
        components: rustfmt, clippy
        
    - name: Cache dependencies
      uses: actions/cache@v3
      with:
        path: |
          ~/.cargo/registry
          ~/.cargo/git
          target
        key: ${{ runner.os }}-cargo-${{ hashFiles('**/Cargo.lock') }}
        restore-keys: |
          ${{ runner.os }}-cargo-
          
    - name: Run tests
      run: cargo test --verbose
      
    - name: Check formatting
      run: cargo fmt -- --check
      
    - name: Run clippy
      run: cargo clippy -- -D warnings

GitLab CI

# .gitlab-ci.yml
image: rust:latest

variables:
  CARGO_HOME: $CI_PROJECT_DIR/cargo

cache:
  paths:
    - cargo/
    - target/

stages:
  - test
  - deploy

test:
  stage: test
  script:
    - rustc --version && cargo --version
    - cargo test --verbose
    - cargo clippy -- -D warnings
    - cargo fmt -- --check

deploy:
  stage: deploy
  script:
    - cargo publish --token $CRATES_IO_TOKEN
  only:
    - tags
